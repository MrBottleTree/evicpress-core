# EvicPress: Technical Reference Summary

## 1. Core Philosophy: Joint Optimization
Unlike traditional systems that perform eviction (LRU) and compression (fixed ratio) independently, EvicPress treats them as a **joint optimization problem**. It acknowledges that some contexts are highly sensitive to compression (needing fast storage/eviction), while others can be aggressively compressed without quality loss (staying in fast storage).

## 2. The Unified Utility Function
The heart of EvicPress is a utility function that quantifies the trade-off between **generation quality** and **loading delay (TTFT)**.

$$Util(\text{method, ratio, device}) = (\alpha \cdot \text{Quality} - \text{TTFT}) \cdot \text{Frequency}$$

*   **$\alpha$ (Alpha):** A tunable parameter. High $\alpha$ prioritizes quality; low $\alpha$ prioritizes low latency.
*   **Quality:** Measured by cosine similarity between compressed and uncompressed KV embeddings.
*   **TTFT (Time-to-First-Token):** Estimated based on the bandwidth of the storage tier and the compressed size of the block.
*   **Frequency:** How often the specific context is accessed (to prioritize "hot" data).

## 3. Storage Hierarchy & Tiering
EvicPress manages KV caches across a multi-tier system:
*   **Tier 1 (Compute Node RAM):** Fastest tier, managed via static partitioning.
*   **Tier 2 (Storage Node RAM):** Intermediate tier for high-speed gRPC retrieval.
*   **Tier 3 (Storage Node Disk/SSD):** Archival storage for low-utility or highly compressed blocks.

**Key Insight:** Blocks on Disk (Tier 3) require **higher compression ratios** than those in RAM to compensate for the significantly lower read bandwidth of SSDs.

## 4. System Workflow

### A. Initial Profiling (Offline Phase)
*   The system uses representative queries (e.g., generated by GPT-5) for each context.
*   It profiles all possible **compression-eviction configurations** (method $\times$ ratio $\times$ tier).
*   It builds a local "Utility Score Directory" for each context.

### B. Online Inference Phase
1.  **Lookup:** Check if the KV block ID exists in the metadata directory.
2.  **Retrieve:** Fetch the block from the indicated tier.
3.  **Store & Manage:** If a new block is generated or a tier is full, the `manage()` function is triggered. It treats placement as a **Multi-Choice Knapsack Problem**.

### C. Periodic Re-profiling
*   **Trigger:** Triggered if achieved quality drops below a threshold $X\%$ or during idle GPU cycles.
*   **Action:** Re-runs profiling with the latest real-world queries to correct "semantic drift" in utility scores.

## 5. The Decision Algorithm (Greedy Knapsack)
When a storage tier $S$ is full, the **Policy Optimizer** runs:
1.  Calculate the "utility drop" for every possible operation (further compression or eviction to a lower tier).
2.  Greedily select the operation with the **minimum utility drop**.
3.  Repeat until the tier size $S_T \le C$ (Capacity).
4.  If the lower tier also becomes full, apply the same logic recursively.

## 6. Implementation Hooks (vLLM + LMCache)
The implementation (approx. 3K lines of Python) hooks into:
*   `LMCache.lookup(tokens)`: Returns hit count and triggers retrieval.
*   `LMCache.retrieve(tokens)`: Loads from the multi-tier backend into vLLM's paged memory.
*   `LMCache.store(tokens, KV_Cache)`: Saves to Tier 2/3.
*   `EvicPress.manage()`: Core logic that executes the Greedy Knapsack algorithm to rearrange tiers.

## 7. Compression Methods Supported
EvicPress is method-agnostic but was evaluated using:
*   **Keydiff:** Cosine-similarity-based token dropping.
*   **Knorm:** L2-norm-based key cache pruning.
*   **SnapKV:** Cross-attention-based token selection.

## 8. Performance Targets
*   **Latency:** Up to 2.19x faster TTFT compared to fixed-ratio baselines.
*   **Throughput:** 2.0x–3.6x improvement by maximizing Tier 1/Tier 2 hit rates.
*   **Efficiency:** Uses a metadata-only control plane; the actual payload is only touched when a transfer is required.
